## **Executive Summary**

The phenomenon of software documentation drift—where the implemented physical architecture of a system incrementally diverges from its documented conceptual intent—represents a profound vulnerability in the lifecycle of large-scale software engineering. As continuous integration, microservice decomposition, and decentralized deployment models accelerate the velocity of code changes, static natural language artifacts, such as Architecture Decision Records (ADRs) and high-level system requirements, frequently fall out of synchronization with the underlying physical code topology. The prevailing academic consensus indicates that traditional methods of code summarization, heuristic-based static analysis, and manual peer review are insufficient for maintaining architectural coherence at scale. Furthermore, the application of traditional, linear Large Language Models (LLMs) to this specific problem domain has revealed fundamental computational and architectural limitations. These models suffer from a phenomenon defined in the literature as "structural blindness"; they treat multi-dimensional, hierarchically complex codebases as flattened, one-dimensional token sequences, thereby failing to grasp global dependencies, spatial execution paths, and deep structural contradictions.

To bridge the semantic and topological gap between human-authored architectural intent and deterministic execution paths, contemporary computer science research has pivoted aggressively toward spatial artificial intelligence, structural algorithms, and Graph Neural Networks (GNNs). By representing codebases as heterogeneous, non-Euclidean graphs—specifically Code Property Graphs (CPGs) that synthesize Abstract Syntax Trees (ASTs), Control Flow Graphs (CFGs), and Data Flow Graphs (DFGs)—GNNs can computationally map abstract intent to discrete topological structures. Emerging academic frameworks demonstrate that graph-aware AI can detect Design-Implementation-Documentation (DID) drift with unprecedented precision.

This report provides an exhaustive synthesis of the state-of-the-art academic literature on these advanced methodologies. It details how high-level architectural decisions are ingested as computable topological constraints, how version control histories are continuously parsed using advanced algorithms to pinpoint the exact emergence of structural decay, and the computational limits inherent in aligning conceptual design with rigid structural topologies. By traversing the intersection of Mining Software Repositories (MSR), artificial intelligence, and software architecture, this document establishes a comprehensive framework for understanding the automated maintenance of structural software integrity.

## **Abstract Intent: The Computational Representation of Human Architectural Decisions**

The foundational challenge in the automated maintenance of software architecture lies in translating abstract, human-readable design documentation into machine-verifiable programmatic constraints. Architectural intent is not merely a description of what a system does; it is a prescriptive mandate regarding how a system must be structurally organized to achieve specific non-functional requirements, such as scalability, security, or maintainability.

### **The Evolution of Architecture Decision Records (ADRs)**

Historically, software architecture was documented in monolithic design specifications that were highly susceptible to obsolescence. In modern agile and distributed paradigms, architectural decisions are predominantly recorded in Architecture Decision Records (ADRs). An ADR is a localized, version-controlled document that captures an important architectural decision, including its context, the decision drivers, the considered options, and the anticipated consequences.1 The collection of these records forms an Architecture Decision Log (ADL), which serves as the authoritative, historical rationale for a project's structural evolution.

However, because ADRs exist primarily as natural language artifacts (often formatted as Markdown files), they exist independently of the executable codebase. They lack intrinsic enforcement mechanisms, leading to a breakdown in traceability. When a developer introduces a new library that violates a previously established ADR, the compiler does not fail, and traditional continuous integration (CI) pipelines do not flag the infraction.3 To computationally represent human architectural decisions, research has increasingly focused on extracting structural intent from these text files and mapping it to structured, queryable intermediate representations.

### **Encoding Semantic Intent into Topological Constraints**

Recent literature proposes transitioning from static specifications toward "living" or executable specifications by linking natural language requirements to graph-based knowledge systems. In this advanced paradigm, an ADR is ingested by a natural language processing pipeline that extracts key architectural entities (e.g., modules, microservices, databases, design patterns) and the permissible topological relationships between them.4

To achieve this, researchers utilize Information Extraction (IE) pipelines enhanced by Large Language Models to parse ADRs and Pull Request (PR) descriptions into formal domain models. These models are mapped into Knowledge Graphs (KGs), where nodes represent architectural components and edges represent data flows, dependencies, or invocation protocols.5 By transforming text into nodes and edges, the abstract intent is given a spatial and mathematical dimension. For instance, the conceptual intent of "Centralized Dependencies" outlined in a standard ADR can be computationally codified into a directed graph where a central "Core" node holds outbound edges to "Feature" nodes, strictly prohibiting cross-communication or cyclic dependencies between the feature nodes themselves.4

### **Diagnosing Structural Contradictions in System Design**

When architectural intent is computationally formalized into a graph, it becomes possible to systematically detect "structural contradictions." A structural contradiction occurs when the optimization pressures, automated code generations, or incremental manual updates within a system slowly violate the foundational rules outlined in the architectural intent.6

In complex software engineering, these contradictions do not necessarily manifest as immediate functional bugs or compilation errors; rather, they appear as "priority inversions," "constraint erosions," or "architectural debt".7 For example, if an ADR dictates a strict separation of concerns between business logic and data access, but successive commits begin interleaving database querying logic within user interface components, a structural contradiction has emerged. The system may still pass all functional unit tests while simultaneously degrading its structural integrity.

Academic models addressing this issue categorize contradictions into distinct taxonomies. While factive or lexical contradictions deal with semantic disagreements in documentation, structural contradictions represent a physical divergence in the anticipated execution paths or structural hierarchies of the system.9 Advanced AI systems designed for structural analysis, such as conceptualized "Reality Compilers," evaluate the evolving graph representation of the physical codebase against the established Knowledge Graph of the ADRs to flag these contradictions before they compound into systemic, unrecoverable failures.7 This allows engineering teams to identify the exact moments when a system drifts away from the very intent that justified its initial design.

## **Managing Documentation Drift: AI and Graph-Based Approaches to Measuring System Decay**

Documentation drift is the quantifiable delta between the expected state of a software system—as defined by its governing documentation, requirements, and architectural diagrams—and its actual, physical implementation in source code.10 Left unmanaged, this drift results in severe productivity losses, architectural erosion, and "dependency blindness" within complex distributed systems.11 Measuring and managing this drift requires sophisticated computational techniques capable of reconciling human language with machine logic.

### **The Mechanics of Design-Implementation-Documentation (DID) Drift**

The academic community frequently categorizes this phenomenon of architectural decay as Design-Implementation-Documentation (DID) drift.10 The drift occurs because maintenance tasks, hotfixes, and iterative feature developments performed on the codebase are rarely fully or accurately translated back into the informal or formal documentation.14 Over time, the documentation ceases to be a reliable source of truth, transforming instead into a historical artifact of what the system *used* to be.

To manage DID drift, automated methodologies have been developed to continuously map and measure the divergence. Prominent approaches utilize visual exploration tools and dynamic metrics to capture documentation coverage evolution across releases. Tools such as *Drifter* have been engineered to extract entities from text-based architecture diagrams (e.g., PlantUML, Mermaid) and map them directly to corresponding source code entities, such as Java classes or Python modules.10 By computing an optimal alignment between the text-based design specifications and the underlying Abstract Syntax Tree (AST), these systems generate quantitative metrics representing the exact degree of drift, allowing architects to visualize the erosion of system intent over discrete Git commits.14

### **Structural Blindness: The Theoretical Limitations of Linear Large Language Models**

A critical consensus in recent empirical literature is the profound limitation of traditional, linear Large Language Models (LLMs) in accurately detecting and managing structural documentation drift. While standard LLMs excel at localized code generation, syntax correction, and boilerplate documentation generation, they fail significantly when tasked with understanding spatial software architecture, global topological properties, or complex dependency graphs.16

This failure is rooted in the fundamental architecture of the Transformer model. Linear LLMs process code and documentation as flattened, one-dimensional sequences of text tokens.17 They calculate attention weights based on probabilistic token proximity rather than logical execution paths. Consequently, they lack the inherent inductive bias required to grasp multi-dimensional spatial relationships, such as deep inheritance hierarchies, polymorphic dispatch paths, or intricate microservice dependency chains.3

The academic community refers to this phenomenon as "structural blindness".17 Because linear models cannot naturally traverse a Control Flow Graph (CFG) or a Data Flow Graph (DFG), they struggle to align physical code with high-level topological maps. When presented with NP-hard graph tasks or repository-scale reasoning requirements, linear LLMs frequently produce hallucinations, logically inconsistent behaviors, or suggest code completions that violate established architectural boundaries.16

The computational overhead required for linear models to simulate graph understanding is also immense. In a linear narrative, understanding whether a specific user can access a specific database through a complex chain of microservices requires the LLM to process massive token contexts, leading to quadratic computational costs ($O(N^2)$) due to the self-attention mechanism.21 For enterprise-scale repositories, this rapid exhaustion of the context window renders linear LLMs impractical for global architectural validation.22

### **Overcoming Blindness with Graph Neural Networks and Spatial AI**

To overcome structural blindness, state-of-the-art research has shifted toward Graph-Enhanced LLMs and pure Graph Neural Networks (GNNs) that inject explicit topological data into the analysis pipeline. GNNs are specifically engineered to process non-Euclidean data where entities (nodes) and their interdependencies (edges) are the primary units of analysis.23

GNNs operate on the principle of message passing, where a node iteratively updates its representation by aggregating features from its structural neighborhood.23 This allows the model to inherently understand the topology of a software architecture. By replacing flat text inputs with structural encodings—such as Laplacian eigenvectors, spectral gradients, or random walk matrix properties—GNNs maintain spatial awareness of exactly where a specific function, class, or module sits within the broader architectural ecosystem.24

The comparative advantages of Graph-Aware AI over Linear LLMs in the context of software architecture are stark, as illustrated by the prevailing academic literature:

| Dimension of Analysis                | Linear Large Language Models (LLMs)                                                            | Graph Neural Networks (GNNs) & Spatial AI                                                             |
| :----------------------------------- | :--------------------------------------------------------------------------------------------- | :---------------------------------------------------------------------------------------------------- |
| **Data Representation**              | 1D sequential token arrays; treats code identically to natural language prose.17               | Non-Euclidean graphs (nodes, edges, attributes); natively maps ASTs and dependency networks.23        |
| **Spatial/Structural Understanding** | Highly limited; relies on textual proximity and probabilistic pattern matching.25              | Native; utilizes message passing to recursively aggregate structural neighborhood contexts.23         |
| **Architectural Drift Detection**    | Prone to hallucinations on global tasks; cannot traverse deep dependency chains reliably.16    | High accuracy; compares implemented code property graphs directly against intended topology graphs.27 |
| **Computational Complexity**         | High resource consumption ($O(N^2)$ attention mechanism) across massive repository contexts.21 | Efficient localized inference ($O(E)$ or linear cost) depending on positional encoding strategy.21    |
| **Primary Optimal Use Case**         | Unstructured text generation, local code completion, heuristic summarization.21                | Structural anomaly detection, cross-file impact analysis, semantic vulnerability mapping.11           |

Hybrid architectures that pair the semantic fluency of LLMs with the topological rigor of GNNs currently represent the vanguard of automated drift detection.21 In these systems, an LLM might parse the unstructured text of an ADR to define the structural rules, while a GNN continuously scans the physical codebase to ensure the resulting execution paths adhere strictly to those rules.

## **The Ingestion of Attractor States: Methodologies for Parsing ADRs, PRs, and Git Logs into Structural Constraints**

In the context of dynamic software development, a system's architecture is never entirely rigid; it undergoes continuous evolution. However, this evolution must ideally orbit around specific "Attractor States." Borrowed from complex systems theory, an Attractor State in software architecture represents a high-stability structural configuration that minimizes structural contradictions, preserves operational intent, and mitigates thermodynamic cost (i.e., technical debt).29

To ensure a system does not drift away from these attractors, automated maintenance protocols rely heavily on the field of Mining Software Repositories (MSR). This involves combining the temporal data of version control history (e.g., Git) with deep Abstract Syntax Tree (AST) parsing to track the precise evolution of structural constraints.

### **Defining Architectural Attractor States**

An Attractor State is established when the abstract intent documented in ADRs is successfully ingested and translated into a baseline architectural graph.30 As developers introduce new code via Pull Requests (PRs), the system's physical topology naturally shifts. The goal of automated drift detection is not to prevent all change, but to ensure that changes do not fundamentally alter the topological invariants defined by the Attractor State.

If a system's coherence is threatened—for example, if a newly introduced dependency creates a cyclic loop that violates a layered architecture—the AI detects a "coherence restructuring" event. The AI must then identify exactly when, how, and why the system was pulled away from its intended Attractor State.29

### **Mining Software Repositories: The SZZ Algorithm and Exact Commit Pinpointing**

A core requirement for understanding and rectifying architectural drift is identifying the precise temporal moment—the exact Git commit—where the implemented code diverged from the documented intent. Version control systems like Git maintain a directed acyclic graph (DAG) of the system's evolution.32 By analyzing this DAG alongside the code's changing internal structure, researchers can track structural decay over continuous timeframes.

The most prominent methodology for pinpointing these divergence points is rooted in the SZZ algorithm and its modern, advanced implementations (e.g., SZZ Unleashed).34 Originally formulated to locate bug-inducing commits (BICs), the SZZ algorithm operates by taking a known issue, identifying the specific lines of code that were altered to fix it, and then tracing the origin of those lines back through the Git history using git blame functionality to find the exact commit that introduced the anomaly.

When repurposed for architectural drift detection, this methodology undergoes a significant enhancement. Instead of merely tracking textual line modifications, advanced SZZ variants analyze the AST of consecutive commits. This allows the algorithm to isolate purely structural modifications—such as the alteration of a public interface, the injection of a disallowed cross-module dependency, or the bypassing of a core abstraction layer.36

### **Abstract Syntax Tree (AST) Parsing and Change Pattern Extraction**

The automated extraction of architectural intent relies heavily on AST parsing. An AST provides a precise, tree-based syntactic structure of the source code, independent of superficial formatting or stylistic variations.38 By parsing the AST of files within a given commit, systems can extract instances of specific "code change patterns."

For example, researchers have developed techniques that utilize AST properties to separate multiple, distinct developer intentions within a single, tangled commit.37 If a developer submits a commit that simultaneously fixes a bug and introduces an undocumented architectural change (a highly common source of DID drift), the AST parsing algorithm clusters the atomic changes. It separates the localized logic shift from the global structural shift, allowing the automated system to evaluate the architectural modification against the stored ADR constraints.36

By continuously extracting and clustering these AST change patterns, automated systems can pinpoint the exact Pull Request where the physical topology violated the rules defined by the ADR. This continuous parsing and graph differencing treats the repository not as a static artifact, but as a continuous data stream, actively monitoring for both immediate structural violations and slow-moving "concept drift".40

### **Spatial Memory Models for Temporal Tracking**

Maintaining an accurate, continuously updated map of a massive codebase over time introduces significant computational hurdles, particularly regarding "catastrophic forgetting" in AI models. To address this, cutting-edge frameworks incorporate spatial memory models.42

In software architecture maintenance, a spatial memory model allows an AI agent to maintain a persistent, externalized topological database (often a vector database or graph database) of the system's state.22 Rather than relying on the limited context window of an LLM to remember how thousands of microservices interact, the agentic system retrieves the relevant structural graph dynamically. When a new commit alters the architecture, the spatial memory is updated incrementally, allowing the system to track architectural decay over extended time horizons without losing the context of historical ADRs.22

## **The Alignment of Concept and Structure: Topologies, AST Mapping, and Automated Refactoring Pathways**

To physically and computationally align conceptual documentation with the deterministic execution paths of the codebase, automated systems must bridge the semantic gap between high-level design and low-level code. The prevailing academic approach involves constructing multi-layered, highly enriched graphs that capture everything from high-level project configurations down to the lowest-level execution semantics.

### **From ASTs to Code Property Graphs (CPGs)**

At the code level, relying solely on a standard Abstract Syntax Tree (AST) is insufficient for detecting deep architectural drift. While ASTs capture precise syntax, they inherently lack semantic depth and context regarding data flow and execution order.38 Furthermore, large files generate massive, linearized ASTs that are computationally expensive to process using standard attention mechanisms.39

To resolve this, state-of-the-art detection methodologies enrich the AST by fusing it with Control Flow Graphs (CFGs) and Data Flow Graphs (DFGs), creating a comprehensive, multi-dimensional Code Property Graph (CPG).11 The CFG maps all possible execution paths through a program, while the DFG tracks how variables and data states propagate through those paths.

When Graph Convolutional Networks (GCNs) and Graph Attention Networks (GATs) are trained on these enriched, hybrid graphs, they achieve significantly higher accuracy in detecting structural anomalies, cross-code similarities, and architectural violations.38 For instance, a framework like HELIOS directly combats the "structural blindness" of LLMs by explicitly feeding the model a multi-level description of the CFG alongside the raw code. This forces the model to explicitly reason over the topological adjacency list rather than just relying on the textual syntax, dramatically improving the detection of logical contradictions.17

### **The Repository Intelligence Graph (RIG) and SPADE Extractor**

While CPGs handle code-level topology, modern software systems are composed of multiple files, modules, and external dependencies. Standard LLM agents frequently fail in multi-file, repository-level tasks because they lack a deterministic mental model of the project's macro-architecture—including build systems, external packages, and test wiring.46

To resolve this macro-level mapping, researchers introduced the Software Program Architecture Discovery Engine (SPADE) designed to construct a Repository Intelligence Graph (RIG).46 The RIG is a deterministic, evidence-backed architectural map. It represents the entire repository as a network of buildable components, test runners, aggregators, and package managers connected by explicit dependency and coverage edges.48

By providing a structured, JSON-accessible view of the RIG to an AI agent, the agent is freed from having to heuristically guess the project's topology through repeated, error-prone tool calls.46 It can reliably trace how a high-level requirement documented in an ADR maps through a build aggregator, down through a specific microservice dependency, and into a specific unit test.

### **The GRACE Framework and Mitigating Structural Information Loss**

A major challenge in mapping intent to code using modern AI is "structural information loss." Traditional Retrieval-Augmented Generation (RAG) approaches—frequently used to give LLMs access to codebases—retrieve code snippets as flat text chunks based on semantic similarity. In doing so, they strip away the complex hierarchical and semantic relationships that the code possessed in its original environment.49

The Graph-Guided Repository-Aware Code Completion (GRACE) framework was explicitly developed to solve this structural information loss. GRACE constructs a multi-level, multi-semantic hybrid code graph representing the entire repository.49 Instead of simply retrieving text strings, GRACE utilizes a Hybrid Graph Retriever to extract the optimal subgraph from the repository. It then deeply fuses this retrieved subgraph with the local query graph both semantically and structurally.49

This graph-aware reranking mechanism ensures that any automated refactoring pathways or architectural validations generated by the AI strictly adhere to the overarching topological constraints of the project. If an AI agent suggests an automated code remediation to fix a drifted module, the GRACE framework ensures the proposed fix does not inadvertently introduce a new, illegal dependency edge in the global graph.

### **Computational Limits and Theoretical Counter-Arguments**

Despite the significant theoretical advancements in spatial AI and graph-aware software maintenance, the academic consensus acknowledges several profound computational limits, algorithmic constraints, and theoretical counter-arguments regarding the automated alignment of conceptual design with deterministic code topologies.

#### **The Limits of Graph Neural Network Expressiveness**

While GNNs vastly outperform linear LLMs in spatial reasoning, they are bound by rigorous mathematical limitations. The expressive power of standard message-passing GNNs is theoretically capped by the Weisfeiler-Lehman (WL) graph isomorphism test.52 Consequently, standard GNN architectures can be "oblivious" to prominent sub-structures, such as complex cyclic dependencies or long-range bottlenecks, which are common in deteriorating, tightly coupled legacy software architectures.52

To detect these highly complex architectural anti-patterns, researchers are forced to utilize Higher-Order Graph Neural Networks (HOGNNs) or incorporate topological data analysis layers (e.g., TOGL) that leverage persistent homology to capture multi-scale topological features.52 However, injecting these global topological features introduces severe computational bottlenecks. Advanced positional encoding methods that rely on pairwise distance matrices or all-node attention scores scale quadratically ($O(N^2)$). When applied to massive enterprise repositories containing tens of millions of nodes (functions, variables, files), this computational overhead becomes prohibitive.24

#### **The Semantic Gap and Scalability Bottlenecks**

A prevailing counter-argument in the empirical literature highlights the enduring "semantic gap" between informal natural language documentation and rigidly formal code structures. High-level architectural descriptions (ADRs) often contain constructive ambiguities, implicit domain knowledge, and factive contradictions that cannot be easily mapped to a strict graph schema without extensive human intervention.54 Automated methods struggle to quantify nuances where an ADR states a service "should generally avoid" a dependency versus "must never invoke" a dependency.

Furthermore, achieving strict formal alignment requires significant computational resources. Frameworks that attempt to dynamically maintain and update massive knowledge graphs in real-time face critical scalability bottlenecks. Revalidating the entire topological graph of a system every time an API spec changes or a minor PR is pushed is highly resource-intensive and prone to high false-positive rates.45

As a result, the academic consensus suggests that fully autonomous, zero-touch architectural alignment remains a theoretical frontier.56 The current optimal paradigm is a hybrid, "co-architect" approach: utilizing lightweight, heuristic linear models for immediate local code assistance, while deploying heavy, computationally intense graph-aware algorithms as asynchronous, continuous integration gatekeepers to periodically enforce structural constraints and prevent long-term documentation drift.45
